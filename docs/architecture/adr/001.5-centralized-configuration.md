# ADR 001.5: Centralized Configuration

**Date:** 2026-05-30
**Status:** Accepted
**Authors:** Robert Capps
**Parent:** ADR 001 (Code Architecture Standards)

---

## Executive Summary

Configuration and canonical constants — environment variables, feature flags,
identifiers, pricing, tier definitions — live in **one canonical location per concern**.
Downstream code reads from that source; it never hardcodes a value or keeps a local
copy. This prevents the drift that occurs when the same setting is read and redefined in
many places.

## 1. Problem Context

### The Problem

When environment variables are read directly wherever they're needed and constants are
redefined per consumer, settings drift. Two modules disagree about a limit; a renamed
variable is updated in some places but not others; a magic number's meaning is lost. The
system's behavior depends on configuration nobody can see in one place.

### Requirements

1. Each setting and canonical constant is defined once, in a known location.
2. Downstream code reads from the canonical source — never a hardcoded literal or local copy.
3. Required configuration is discoverable, documented, and validated at startup.

### Scope

Covers configuration and canonical constants. Deduplication of logic is 001.1; this is
its application to settings and values.

---

## 2. Decision: One Canonical Source Per Concern

### Decision

Every concern has a single canonical configuration source — a config module that reads
and validates the environment, a single table of canonical constants, one definition of
each flag. Consumers import from the canonical source. Hardcoded URLs, secrets, magic
numbers, and direct ad-hoc environment access in arbitrary modules are defects.

When configuration code is refactored, every setting the old code consumed must still be
consumed by the new code — a silently dropped setting changes behavior without warning.

### Alternatives Considered

| Alternative                  | Description                                  | Why Rejected                                          |
| ---------------------------- | -------------------------------------------- | ----------------------------------------------------- |
| Direct environment access everywhere | Each module reads env vars itself     | No single source; renames and defaults drift           |
| Hardcoded constants per consumer | Inline the value where it's used         | Same value redefined and changed inconsistently        |
| Multiple "config" modules    | Several overlapping config locations         | Ambiguous source of truth; the drift problem returns    |

### Rationale

1. One definition per setting means one place to change and to audit.
2. A central source can validate required configuration at startup and document it.
3. Reading from the canonical source keeps consumers consistent by construction.

### Enforcement

Per project: a single config module/namespace per concern; lint forbidding direct
environment access or hardcoded secrets outside it; an example environment file and
documentation kept in sync; a review step that compares old vs. new consumed settings on
any config refactor.

---

## 3. Conclusion

Centralized Configuration prevents drift across the dependency graph: each setting and
constant has one home, downstream code reads from it, and required configuration is
validated and documented in one place. It is DRY (001.1) applied to configuration.
