# ADR 001.4: Interface Boundaries

**Date:** 2026-05-30
**Status:** Accepted
**Authors:** Robert Capps
**Parent:** ADR 001 (Code Architecture Standards)

---

## Executive Summary

A unit exposes a **contract** — types, interfaces, factory functions — and hides its
implementation. Consumers depend on the contract, never on internal file paths or
unexported symbols. The contract is the boundary: it is what can be relied upon, what
can be mocked for testing, and what must stay stable while the implementation behind it
is free to change.

## 1. Problem Context

### The Problem

When consumers import a unit's internals, every internal becomes a de-facto public API.
Refactors break distant code, abstractions leak, and units cannot be tested in
isolation because their dependencies cannot be substituted. The boundary the author
intended is not the boundary the code enforces.

### Requirements

1. A unit's public surface is explicit and minimal; internals are unreachable from outside.
2. Consumers depend only on the contract, so internal refactors are safe.
3. Units are testable in isolation by substituting contracts with test doubles.

### Scope

Covers the contract surface of a unit and of a service. Dependency direction is 001.2.

---

## 2. Decision 1: Units Expose Contracts, Not Implementations

### Decision

Each unit declares an explicit public surface — the types, interfaces, and factory
functions it supports — and keeps everything else private. Consumers import the
contract; they never reach past it to internal paths or unexported symbols. Function
signatures accept interfaces where substitution matters, so collaborators can be mocked.

### Alternatives Considered

| Alternative                 | Description                                | Why Rejected                                       |
| --------------------------- | ------------------------------------------ | -------------------------------------------------- |
| Export everything           | All symbols are importable                 | Every internal becomes public; refactors break consumers |
| Implicit boundary by convention | "Please don't import internals"        | Unenforced conventions decay; the compiler can't help |

### Rationale

1. A minimal public surface lets the implementation change without breaking consumers.
2. Contracts make units testable in isolation.
3. A small surface is easier to review, document, and keep stable.

### Enforcement

Per project: a declared public entry point (package `exports`/visibility, internal
packages, module boundaries); import restrictions that forbid reaching into internals.

---

## 3. Decision 2: The Contract Is the Boundary Between Services

### Decision

Where units communicate across a process or service boundary, a declared contract (an
API schema, an interface definition) is the boundary. Services depend on the contract,
not on each other's data stores or internal structures. Cross-cutting access to shared
state goes through the owning unit's contract, not around it.

### Alternatives Considered

| Alternative                    | Description                                  | Why Rejected                                              |
| ------------------------------ | -------------------------------------------- | -------------------------------------------------------- |
| Shared database reach-through  | Services read each other's tables directly   | Couples services to one another's storage; no contract    |
| Undocumented ad-hoc endpoints  | Call whatever exists, no declared schema      | The boundary drifts; consumers break without warning      |

### Rationale

1. A declared contract is the unit of compatibility — it can be versioned and validated.
2. Going through contracts preserves each unit's freedom to change its internals/storage.

### Enforcement

Per project: an authored API/contract spec as the inter-service boundary; runtime
validation against it; guards forbidding direct access to another unit's storage.

---

## 4. Conclusion

Interface Boundaries protect each node in the dependency graph (001.2): a small,
explicit contract is the only thing consumers may rely on, leaving implementations free
to evolve. Contracts also make isolation testing possible and define the seam between
services.
