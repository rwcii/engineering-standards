# ADR 001.3: Extensible by Construction

**Date:** 2026-05-30
**Status:** Accepted
**Authors:** Robert Capps
**Parent:** ADR 001 (Code Architecture Standards)

---

## Executive Summary

Systems grow by adding variants — a new provider, object type, channel, or tier.
Architecture should make adding a variant a matter of *implementing a contract and
registering it*, not editing a dozen scattered dispatch sites. Registries,
polymorphism, and lookup tables are preferred over switch/if-else chains keyed on a
known set of types.

## 1. Problem Context

### The Problem

When behavior is dispatched with `switch`/`if-else` chains on a type discriminator,
every new variant requires finding and editing every chain. Sites are missed, defaults
fall through silently, and the change's cost grows with the system. The shape of the
code resists the growth the product demands.

### Requirements

1. Adding a variant touches as few existing files as possible — ideally one new file plus one registration.
2. Missing a dispatch site is structurally impossible, not a matter of diligence.
3. The set of variants is discoverable from one place.

### Scope

Covers extension mechanisms for variant families. Contract design is 001.4; collapsing
duplicate per-variant code is 001.1.

---

## 2. Decision: Prefer Registries and Polymorphism Over Hardcoded Dispatch

### Decision

Behavior that varies by a known set of types is expressed through a registry, a
polymorphic contract, or a lookup table — not through dispatch chains repeated at call
sites. A new variant implements the shared contract and registers itself; existing code
is not modified to accommodate it.

### Alternatives Considered

| Alternative                     | Description                                   | Why Rejected                                              |
| ------------------------------- | --------------------------------------------- | -------------------------------------------------------- |
| Switch/if-else dispatch chains  | Branch on a type discriminator at each site   | Adding a variant edits N sites; missed sites are silent bugs |
| One mega-handler per concern    | A single function that knows all variants     | Becomes a coupling point; grows without bound             |

### Rationale

1. Adding a variant is additive: implement, register, done.
2. A registry is the single discoverable list of variants.
3. This is the structural counterpart to DRY's factory/registry mechanism (001.1, Decision 3).

### Enforcement

Per project: a registry or polymorphic dispatch as the extension point; lint
(`no-restricted-syntax` or equivalent) flagging branch-on-known-type-set where a
registry should be used; review that rejects new hardcoded dispatch.

---

## 3. Conclusion

Extensible by Construction keeps the dependency graph (001.2) open to growth: variants
attach at a registration point instead of forcing edits across existing code. The
system accommodates new requirements by addition, not by modification.
