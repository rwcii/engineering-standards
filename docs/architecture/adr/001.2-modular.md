# ADR 001.2: Modular Architecture

**Date:** 2026-05-30
**Status:** Accepted
**Authors:** Robert Capps
**Parent:** ADR 001 (Code Architecture Standards)

---

## Executive Summary

Each unit of the system — package, module, or service — has one well-defined
responsibility, dependencies flow in a single direction, and no unit reaches into
another's internals. This keeps change local and predictable: a modification touches
the unit that owns the concern and propagates only through declared contracts.

## 1. Problem Context

### The Problem

Without explicit module boundaries, responsibilities smear across units and
dependencies form cycles. A change to one unit then cascades unpredictably because
callers depend on internals rather than contracts, and there is no single owner for a
given concern.

### Requirements

1. Each concern has exactly one owning unit.
2. Dependencies form a directed acyclic graph; lower layers never depend on higher layers.
3. A change's blast radius is bounded by declared contracts, not hidden coupling.

### Scope

Covers unit responsibility and dependency direction. Contract design is 001.4;
deduplication is 001.1.

---

## 2. Decision 1: One Responsibility Per Unit

### Decision

Every unit has a single, stated responsibility. Code lands in the unit that owns its
concern; a change that doesn't fit any unit's responsibility signals a missing unit,
not a place to overload an existing one.

### Alternatives Considered

| Alternative          | Description                          | Why Rejected                                       |
| -------------------- | ------------------------------------ | -------------------------------------------------- |
| Catch-all "utils" unit | A grab-bag module for anything      | Becomes a coupling magnet with no clear ownership  |
| Responsibility by convenience | Put code where it's easiest to call | Erodes boundaries; ownership becomes ambiguous     |

### Rationale

1. Clear ownership makes the right home for a change obvious.
2. Single-responsibility units are easier to test and reason about in isolation.

### Enforcement

Per project: a documented responsibility per unit; review that rejects changes placed
outside their owning unit.

---

## 3. Decision 2: Acyclic, One-Directional Dependencies

### Decision

Dependencies flow one way: higher-level units depend on lower-level units, never the
reverse, and never in a cycle. A unit depends on another's **contract**, never its
internal files or unexported symbols.

### Alternatives Considered

| Alternative              | Description                              | Why Rejected                                          |
| ------------------------ | ---------------------------------------- | ----------------------------------------------------- |
| Bidirectional dependencies | Units freely import each other          | Cycles make units inseparable and untestable          |
| Reaching into internals  | Import another unit's private files       | Implicit coupling the compiler cannot protect          |

### Rationale

1. An acyclic graph can be built, tested, and reasoned about layer by layer.
2. Depending only on contracts means internal refactors don't break consumers.

### Enforcement

Per project: dependency-direction checks (layering lint, visibility/`exports`
boundaries, import restrictions); CI guard that fails on a cycle or a cross-internal import.

---

## 4. Conclusion

Modular Architecture defines the dependency graph: one responsibility per node, edges
in a single direction, and every edge landing on a contract rather than an internal.
It is the structure that Centralized Configuration (001.5) and Interface Boundaries
(001.4) build on.
