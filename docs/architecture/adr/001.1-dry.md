# ADR 001.1: DRY — Don't Repeat Yourself

**Date:** 2026-05-30
**Status:** Accepted
**Authors:** Robert Capps
**Parent:** ADR 001 (Code Architecture Standards)

---

## Executive Summary

Every piece of knowledge — a business rule, a type definition, a validation rule, a
constant — should have a single authoritative representation. Duplication across unit
boundaries drifts apart over time, producing inconsistent behavior and bugs fixed in
one place but not another. This ADR codifies three mechanisms that keep a codebase
DRY, stated independently of any language or toolchain.

## 1. Problem Context

### The Problem

A multi-unit architecture pressures contributors to duplicate logic at boundaries.
When several units need the same type definitions, validation rules, or normalization
logic, copying is the path of least resistance — and the copies drift. Inconsistent
error messages, conflicting "truth," and bugs that reappear after being fixed all
follow from duplicated knowledge.

### Requirements

1. A single authoritative location for every business rule, validation rule, and type.
2. Adding or changing a rule requires editing exactly one place.
3. Safety across boundaries — a contract change should break consumers visibly, not silently.
4. No hand-written duplicate of anything that can be generated from a source of truth.

### Scope

This ADR covers deduplication strategy. It does not cover unit decomposition
(see 001.2) or extension mechanisms (see 001.3).

---

## 2. Decision 1: One Authoritative Home for Each Piece of Knowledge

### Decision

Business logic, validation rules, and shared types live in a single canonical unit
(a shared module/package/library). Consumers import from it; they do not reimplement.
When a consumer needs normalization, matching, or validation, it depends on the
canonical unit rather than writing a local copy.

### Alternatives Considered

| Alternative                  | Description                              | Why Rejected                                          |
| ---------------------------- | ---------------------------------------- | ----------------------------------------------------- |
| Per-unit business logic      | Each unit owns its own rules             | Drift between units; bugs fixed in one, not others    |
| Shared via copy-paste/sync   | Periodically sync files between units    | No compile-time enforcement; syncs fall behind quickly |
| Shared logic as a network service | Expose in-process rules over the network | Latency and availability coupling; overkill            |

### Rationale

1. A single source means one rule to change, not N copies to find.
2. Build- or compile-time resolution surfaces breakage at the boundary instead of in production.
3. Consumers stay thin and consistent.

### Enforcement

Per project: a shared module/package consumed by others; lint or guards that flag
re-implementation of canonical logic; build ordering that makes shared units a
prerequisite of their consumers.

## 3. Decision 2: Generate, Don't Hand-Duplicate

### Decision

Anything derivable from a single source of truth — HTTP contract types, schema types,
client SDKs — is **generated** from that source, not hand-written in each consumer.
The generated artifact is committed so diffs are reviewable, and runtime validation
derives from the same source as compile-time types.

### Alternatives Considered

| Alternative                   | Description                               | Why Rejected                                              |
| ----------------------------- | ----------------------------------------- | -------------------------------------------------------- |
| Hand-written duplicate types  | Maintain types by hand in each consumer   | Drift guaranteed; N places to update per change           |
| Runtime-only validation       | Validate at runtime, untyped at build     | Loses build-time safety; bugs surface in production        |

### Rationale

1. One source produces all derived artifacts — add a field once.
2. Generated output is deterministic and reviewable.
3. Runtime validation and static types agree because they share a source.

### Enforcement

Per project: a codegen step in the build, generated files committed, and a check that
fails when generated output is stale relative to its source.

## 4. Decision 3: Collapse Variant Families Behind a Factory or Registry

### Decision

Where many implementations share a shape — providers, adapters, drivers — route them
through a single contract via a factory or registry. Consumers request the contract
and receive a variant-agnostic instance; variant-specific quirks are encapsulated
behind the factory, not duplicated at every call site.

### Alternatives Considered

| Alternative                      | Description                            | Why Rejected                                         |
| -------------------------------- | -------------------------------------- | ---------------------------------------------------- |
| Direct per-variant usage         | Each consumer configures each variant  | Duplicated config, error handling, retry logic        |
| Conditional logic at call sites  | `if variant == X … else …` everywhere  | Branching scattered; adding a variant touches N files |

### Rationale

1. Adding a variant means implementing the contract and registering it — no other changes.
2. Variant-specific behavior is encapsulated, keeping consumers agnostic.
3. This is DRY applied to behavior, and it directly supports Extensible by Construction (001.3).

### Enforcement

Per project: a factory/registry entry point; lint against direct variant instantiation
in consumer code.

---

## 5. Conclusion

DRY operates at three layers: knowledge lives in one canonical home, derived artifacts
are generated from a single source, and variant families hide behind a single contract.
The cost — build-order dependencies and updating consumers when a shared contract
changes — is accepted because it surfaces breakage at build time rather than in production.
